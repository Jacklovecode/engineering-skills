# Review

## 审查门禁

实现完成后、合并前、重大变更后，必须进行审查。

审查发现测试失败、构建失败、行为异常、性能退化、结果无法解释时，不要直接修，进入 `diagnose`。

## 双轴审查

审查前先明确审查范围。优先使用 git diff、基准分支与当前分支的差异、PR 差异，或用户给出的文件范围。

如果无法确定范围，先说明你基于哪些文件和假设进行审查。

## 审查输入

审查前必须尽量固定三类输入：

- 对比点：提交、分支、标签、PR、当前工作区或用户指定文件。
- 需求来源：用户请求、计划、规格说明、issue、PRD 或设计文档。
- 规范来源：AGENTS/CLAUDE/GEMINI、CONTRIBUTING、CONTEXT、ADR、样式配置或项目约定。

如果缺少需求来源，不要假装知道需求；把需求审查标记为“依据不足”，并只审查能从现有信息确认的范围。

如果平台支持并行子任务，可以把需求审查和工程审查分开执行，再汇总结果；不要让工程偏好掩盖需求遗漏。

需求审查和工程审查必须分开思考、分开记录。不要把“代码写得不错”当成“需求已满足”，也不要把“需求实现了”当成“工程质量可靠”。

### 1. 需求审查

检查实现是否符合用户真正要求的行为。

关注：

- 是否覆盖核心需求
- 是否遗漏边界条件
- 是否实现了未要求的功能
- 是否改变了不该改变的行为
- 是否和计划或设计不一致
- 是否缺少验收证据

### 2. 工程审查

检查实现是否符合工程质量。

关注：

- 测试是否验证外部可观察行为
- 是否按任务分层交付
- 模块边界是否清晰
- 命名是否贴合项目领域语言
- 是否引入不必要复杂度
- 是否存在重复、隐藏耦合、过度使用模拟替身
- 验证命令是否运行并通过
- 是否符合项目已有规范来源

## 问题分级

- 严重：会导致功能错误、数据损坏、安全问题、测试失败、构建失败，必须阻止完成。
- 主要：需求遗漏、明显架构问题、难维护设计，应该修复后再完成。
- 次要：命名、局部重复、小型清理，不阻止完成但应记录。
- 疑问：需要用户或领域知识确认的问题。

## 输出

先列发现，按严重程度排序。没有发现也要说明测试缺口或残余风险。

如果存在严重或主要问题，不要进入 `finish`。

如果没有严重或主要问题，并且验收证据充分，下一步进入 `finish`。

每个发现应包含：

- 位置：文件和行号，或明确的模块/行为
- 问题：具体失败模式或维护风险
- 依据：需求、计划、测试结果或代码证据
- 建议：下一步应该修复、诊断、澄清还是接受风险

输出必须包含：

- 审查范围：对比点和文件范围
- 需求来源：使用了哪些需求、计划或设计依据
- 规范来源：使用了哪些项目规范或约定
- 需求审查结果
- 工程审查结果
- 阻塞项：严重/主要问题是否存在
- 下一步：`diagnose`、`clarify`、修复、接受风险或 `finish`
