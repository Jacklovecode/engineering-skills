# Deepen

## 架构深化检查点

当用户想改进架构、重构模块、降低耦合、提升可测试性，或你在 `diagnose`、`review`、`zoom-out` 中发现结构性摩擦时，使用 `deepen`。

不要在没有理解领域语言、现有决策和调用关系时提出架构方案。

## 核心语言

使用这些词保持判断一致：

- 模块：有接口和实现的单元，可以是函数、类、包、业务单元。
- 接口：调用者必须知道的一切，包括类型、约束、错误模式、顺序、配置，不只是函数签名。
- 实现：模块内部代码。
- 深度：小接口背后承载大量有价值行为的程度。
- 深模块：接口简单、行为丰富、调用者负担小。
- 浅模块：接口复杂度接近实现复杂度，调用者仍要知道太多细节。
- 替换点（seam）：可以替换行为而不在原地硬改的位置。
- 适配器：在替换点实现接口的具体实现。
- 接口收益：调用者通过小接口获得的能力。
- 改动集中度：修改、缺陷和知识是否集中在少数地方。

## 判断方法

### 1. 读取语境

先查看相关代码、测试、`CONTEXT.md`、`CONTEXT-MAP.md`、ADR 和 README。

可使用只读脚本辅助收集架构语境：

```text
skills/deepen/scripts/collect-architecture-context.ps1
skills/deepen/scripts/collect-architecture-context.sh
```

脚本只收集领域文档、ADR、入口文件、测试目录和 Git 差异线索，不替代架构判断。

使用项目已有领域语言描述问题，不要只用文件名或技术层名称。

如果 `CONTEXT.md`、`CONTEXT-MAP.md` 或 ADR 不存在，明确说明缺少哪些语境来源。不要假装已有领域语言；可以从代码、测试、README 中推断候选术语，但必须标注为“推断”。

### 2. 寻找结构性摩擦

重点找：

- 理解一个概念需要在许多小模块之间来回跳转。
- 模块很浅，只是把复杂度转移给调用者。
- 为了测试抽出许多小函数，但真实缺陷仍发生在调用关系里。
- 多个模块互相泄漏内部细节。
- 当前接口难以写行为测试。
- 同一个领域概念散落在多个地方。
- 修改一个行为需要同时改很多无关文件。

### 3. 做删除测试

怀疑某个模块很浅时，问：

- 如果删除这个模块，复杂度会消失，还是会分散到多个调用者？
- 如果只是把复杂度搬家，它没有提供足够接口收益。
- 如果删除后多个调用者都要重复理解同一规则，它可能值得保留或加深。

### 4. 判断替换点是否真实

- 一个适配器通常只是“假设中的替换点”。
- 两个或更多适配器才说明替换点真实存在。
- 不要为了抽象而抽象；只有当替换、测试或隔离真的需要时才引入替换点。

### 5. 形成候选方案

先形成候选，再设计接口。不要一开始就发明新抽象；先证明当前摩擦真实存在，并说明为什么现在值得处理。

每个候选必须包含：

- 涉及文件或模块
- 当前摩擦
- 建议改法
- 预期收益：改动集中度、接口收益、可测试性、智能体可理解性
- 风险和迁移成本
- 推荐强度：强烈、值得探索、暂时观望
- 为什么现在做
- 为什么不是其他候选

如果方案与 ADR 冲突，只在摩擦真实且值得重新讨论时提出，并明确指出冲突。

不要先设计新接口。先提出摩擦和候选，等用户选中候选后再进入 `grill` 细化接口和迁移方案。

如果找不到值得做的架构深化，也要明确说明“暂不建议重构”，并给出原因。

## 输出

默认输出 2-5 个架构深化候选，并给出最推荐先做哪一个。

输出格式：

```text
候选 N：名称
涉及范围：
当前摩擦：
建议改法：
收益：
风险：
推荐强度：
为什么优先/不优先：
```

输出末尾必须包含：

```text
首选建议：
为什么现在做：
为什么不是其他候选：
下一步：
```

不要把所有候选说成同样重要。必须给出排序依据。

## 后续动作

- 用户选中候选后，进入 `grill` 压力测试设计。
- 需要拆实现任务时，进入 `plan`。
- 需要证明行为不变时，进入 `tdd`。
- 如果发现架构导致 bug 难以锁定，回到 `diagnose` 说明根因和测试入口问题。

## 不要做

- 不要直接大规模改代码。
- 不要为了“看起来干净”重构。
- 不要重新争论已有 ADR，除非有真实摩擦。
- 不要只说“拆模块”“抽接口”；必须说明改动集中度、接口收益和测试收益。
