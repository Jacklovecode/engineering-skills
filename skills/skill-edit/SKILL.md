# Skill Edit

## 技能编写检查点

创建或修改技能前，必须先定义压力场景。

压力场景用于验证：

- 没有该技能时，智能体会犯什么错
- 有该技能时，智能体应该怎么做
- 哪些压力会导致智能体跳过流程

如果只是修改错别字或格式，可以直接修改；如果会改变触发条件、约束、流程或输出，必须先定义压力场景。

没有压力场景验证，不要发布技能变更。

技能文档也是可测试产物。不能只读一遍觉得清楚，必须用场景证明它能改变智能体行为。

## 描述规则

`description` 只写触发条件，不写流程摘要。

好的描述回答：“什么时候应该读取这个技能？”

不要让描述替代正文流程。描述如果写了完整流程，智能体可能只按描述行动而不读取正文。

描述必须：

- 使用第三人称语气。
- 包含具体触发场景、症状或任务类型。
- 避免总结步骤、检查表或内部流程。
- 控制长度，便于平台检索和识别。

## 流程

### 1. 定义压力场景

至少写 3 个场景：

- 正常场景：技能应该被触发
- 压力场景：用户催促、任务简单、看似可以跳过
- 冲突场景：多个技能可能适用，或用户指令和技能约束有张力

每个场景必须写清：

- 用户会怎么说
- 没有技能时智能体可能犯什么错
- 有技能时正确行为是什么
- 通过或失败如何判断

### 2. 写技能

技能必须包含：

- 何时使用
- 何时不使用
- 必须遵守的约束
- 具体流程
- 完成标准
- 常见误区

### 3. 验证技能

用压力场景检查：

- 智能体是否能识别该用技能
- 智能体是否按流程执行
- 智能体是否在压力下仍遵守约束
- 智能体是否不会过度触发

验证至少覆盖：

- 正常触发：用户明确提出该类任务。
- 压力触发：用户催促、任务看似简单、容易跳过流程。
- 冲突触发：多个技能都像适用，或用户指令和技能约束有张力。
- 过度触发：相邻但不该使用该技能的任务。

如果验证发现智能体仍会跳过流程、误触发或输出格式错误，先修技能正文，再重新验证。

不要为了一次性通过而把规则范围写得过宽。范围过宽的技能会影响相邻任务。

### 4. 精简技能

删除空话、重复、空泛原则。保留能改变行为的指令。

## 质量标准

一个好技能应该：

- 触发条件明确
- 名字短
- 描述能被检索到
- 约束能阻止常见错误
- 流程能直接执行
- 内容不过度冗长
- 引用材料按需放到 references/

## 自审清单

- 触发条件是否清楚到能被平台检索和识别？
- 正文是否包含会改变行为的约束，而不是空泛建议？
- 是否说明何时停止、回退或转入其他技能？
- 是否避免把流程摘要写进 `description`？
- 是否有常见误区或停止标志？
- 是否能被一个不了解本会话的人独立使用？

## 发布前检查

- frontmatter 元数据必须有 `name` 和 `description`。
- `description` 只写触发条件，不写流程摘要。
- 正文必须能阻止至少一个常见错误。
- 压力场景必须覆盖“任务很简单”“用户很急”“多个技能都像适用”。
- 技能名必须短、清楚，只使用字母、数字和连字符。
- 如果流程依赖外部文件，引用层级不要过深。
- 如果引用外部项目思想，必须在 README 或 README_EN 中保留许可和来源说明。

可使用只读脚本辅助检查技能结构：

```text
skills/skill-edit/scripts/check.ps1
skills/skill-edit/scripts/check.sh
```

脚本只检查结构和元数据，不替代压力场景验证。
