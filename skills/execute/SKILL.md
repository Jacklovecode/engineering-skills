# Execute

## 执行检查点

已有计划后，必须使用 `execute` 执行，不要从计划直接跳到“完成”。

如果计划仍缺少目标、文件、测试、实现路径、验证命令或完成标准，先回到 `plan` 补齐。

如果任务之间依赖不清、边界不清、验收不清，先进入 `grill`，不要边做边猜。

## 执行原则

- 按计划顺序逐项执行。
- 不跳过任务，不合并未完成任务，不悄悄扩大范围。
- 每次只处理一层任务。
- 每个任务都必须经过测试、实现、验证和审查。
- 任务失败时进入 `diagnose`，不要连续猜修。
- 所有任务完成后必须进入 `finish`，不能提前声称完成。

## 单任务循环

对计划中的每一个任务执行：

1. 读取任务目标、预计文件、测试、验证命令和完成标准。
2. 进入 `tdd`：先写失败测试，确认失败原因正确，再写最小实现。
3. 运行该任务要求的验证命令。
4. 进入 `review`：检查需求符合度和工程质量。
5. 如果验证或审查发现严重/主要问题，进入 `diagnose`。
6. 根因修复后，重新运行该任务验证，并再次进入 `review`。
7. 记录任务完成证据，再进入下一个任务。

## 失败处理

出现以下情况时停止当前执行循环，进入 `diagnose`：

- 测试失败但原因不符合预期
- 构建、类型检查、lint 或集成验证失败
- 行为异常、性能退化、结果无法解释
- 审查发现严重或主要问题
- 任务完成标准无法证明

诊断完成后，回到当前任务继续执行，不要跳到后续任务。

## 范围控制

执行中发现新需求、设计分支、架构风险或领域语言冲突时：

- 不要直接实现额外功能。
- 记录发现。
- 必要时回到 `clarify`、`grill` 或 `deepen`。
- 只有用户确认或计划更新后，才把新工作纳入执行。

`execute` 是带停顿点的执行器，不是无条件跑完计划的流水线。计划与现实不一致时，先回退到合适技能修正判断，再继续执行。

## 完成标准

只有当所有计划任务都满足以下条件，才进入 `finish`：

- 每个任务的测试和验证命令有新鲜通过证据
- 每个任务已通过 `review`
- 严重/主要问题已清零
- 没有未处理的诊断项
- 没有超出计划范围的未确认改动

## 输出

执行过程中保持任务状态清晰：

- 当前任务
- 已运行的验证命令和结果
- 审查状态
- 阻塞项或诊断结论
- 下一步任务

最终输出前必须先进入 `finish`。
