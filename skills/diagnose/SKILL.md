# Diagnose

## 诊断检查点

当出现以下任一情况时，必须使用 `diagnose`：

- 测试失败
- 构建失败
- lint 或类型检查失败且原因不明确
- 运行结果不符合预期
- 用户报告 bug
- 性能退化
- 行为偶发失败或不稳定
- `review` 或 `finish` 阶段发现异常
- 你无法解释为什么结果正确

在完成诊断前：

- 不要立刻修补。
- 不要连续尝试多个“可能有用”的改动。
- 不要用症状修复代替根因分析。
- 不要修改多个变量来测试一个猜想。

## 流程

### 1. 建立反馈循环

优先构造一个快速、可重复、智能体可运行的通过/失败信号，也就是本次诊断的反馈循环。

这是诊断的核心。没有反馈循环，不要猜测后直接修改。

可选方式：

- 自动化测试
- 命令行命令
- HTTP/curl 脚本
- Playwright/Puppeteer 浏览器脚本
- 测试夹具或跟踪记录（trace）回放
- 最小复现脚本
- 随机输入或属性测试循环，用于提高偶发错误复现率
- 新旧版本、不同配置或不同数据集的差分对比
- 性能测量脚本
- `git bisect run`
- 人工参与脚本：如果必须人工点击，也要把步骤、输入和观察结果结构化记录下来

如果必须人工参与复现，可使用记录脚本：

```text
skills/diagnose/scripts/hitl-loop.ps1
skills/diagnose/scripts/hitl-loop.sh
```

脚本会把人工步骤、输入、预期结果、实际结果和证据位置记录到 `.local/diagnose/`，之后应尽量把记录转化为自动化反馈循环。

如果无法建立反馈循环，停止并说明尝试过什么，需要用户提供什么。

建立后继续打磨反馈循环：

- 更快：缩小测试范围、减少无关启动和等待。
- 更准：断言具体症状，不只断言“没有崩溃”。
- 更稳：固定时间、随机种子、文件系统、网络和外部依赖。

反馈循环完成标准：

- 快：能在可接受时间内反复运行，优先秒级或单一命令。
- 准：断言用户报告的具体症状，而不是只检查“命令没报错”。
- 稳：同样输入能得到同样结论；偶发问题必须记录复现率，例如 20 次失败 8 次。
- 能变红：修复前能捕获问题，修复后能证明问题消失。
- 智能体可运行：除非确实需要人工，否则不依赖手动点击或主观观察。

如果反馈循环不满足这些标准，不要进入假设验证；先继续收紧反馈循环，或说明缺少什么环境、数据或权限。

如果问题是偶发的，不要求一次干净复现；目标是提高复现率。可以循环触发、并行触发、增加压力、缩小时序窗口，直到足够稳定可诊断。

### 2. 复现问题

运行反馈循环，确认它复现的是用户描述的问题，而不是旁边的另一个问题。

### 3. 提出假设

提出 3-5 个可证伪假设，并按可能性排序。

每个假设必须写清：

- 如果这个假设成立，应该观察到什么？
- 如果这个假设不成立，应该观察到什么？

把假设列表展示给用户。用户可能知道最近变更、线上现象或已排除方向；如果用户暂时不回应，可以按你的排序继续，但必须记录排序依据。

### 4. 验证假设

一次只验证一个假设。

验证不是猜测，也不是直接修补。验证必须通过明确证据完成，例如：

- 断点观察
- 针对性日志
- 性能计时
- trace 对比
- 请求/响应记录
- 性能分析器
- 数据库查询计划
- 新旧版本输出差异
- 最小复现对比

不要到处乱加日志。不要同时修改多个变量。临时日志必须带唯一前缀，例如 `[DEBUG-xxxx]`，修复后用该前缀搜索并清理。

性能问题不要靠大量日志猜测。必须先建立基线测量，再用性能分析器、计时、查询计划或版本对比定位。没有基线，不要声称性能已改善。

如果同一问题连续三次修复失败，或每次修复都暴露新的相邻问题，停止继续局部试修，并必须评估是否进入 `deepen`。重点检查是否存在架构问题：没有合适的测试入口、隐藏耦合、共享状态过多，或当前抽象方向错误。

### 5. 修复根因

找到根因后：

- 先写或更新回归测试
- 确认测试能失败
- 做最小修复
- 确认测试通过
- 重新运行原始反馈循环

回归测试必须落在合适的测试入口上：它要覆盖真实触发路径，而不是只验证一个太浅的内部函数。没有合适的测试入口时，要把这件事作为架构发现记录下来。

### 6. 清理复盘

- 删除临时日志和调试脚本
- 搜索并删除带唯一前缀的临时调试输出
- 记录根因
- 说明为什么修复有效
- 如果没有合适的测试入口，说明架构问题并建议后续改进
- 说明什么预防措施能避免同类问题再次发生

## 输出必须包含

- 反馈循环是什么
- 是否成功复现
- 假设列表
- 验证了哪个假设
- 找到的根因
- 修复方式
- 回归测试或无法测试的原因
- 重新验证结果

## 下一步

- 根因已修复且重新验证通过：进入 `review`，再进入 `finish`。
- 修复后仍失败或结果无法解释：回到本技能的假设和验证阶段。
- 发现缺少合适的测试入口、隐藏耦合或架构方向错误：进入 `deepen`。
- 无法建立反馈循环：停止并向用户说明需要的环境、日志、录屏、数据或权限。
