# Plan

## 计划检查点

当需求、规格说明或设计已经确认，但还没有清晰实现路径时，必须先写计划。

如果任务超过一个简单文件修改，或涉及多个模块、测试、数据流、UI/后端协作，不要直接实现。

如果需求、设计或计划仍有分支、依赖、边界、术语、异常路径或验收标准不明确，先进入 `grill`，不要强行写计划。

## 任务分层

任务分层指的是：按完整、可运行、可验证的小行为拆分任务，而不是按数据库、API、UI 等技术层拆分任务。

错误做法：按技术层横向拆分。

- 先建所有数据库表
- 再写所有 API
- 再写所有 UI
- 最后再测试

正确做法：按可观察行为分层。

- 用户可以用邮箱和密码注册
- 邮箱重复时显示错误
- 密码太短时显示错误

每一层任务都必须包含它需要的最小测试、最小接口、最小实现和最小验证。

## 每个任务必须包含

- 目标：这一层任务交付什么可观察行为
- 文件：预计修改或创建哪些文件
- 接口：本任务依赖什么、产出什么，后续任务依赖什么
- 测试：先写什么失败测试
- 实现：最小实现路径
- 验证：运行什么命令证明它通过
- 预期失败：失败测试第一次运行时应该如何失败
- 预期通过：实现后验证命令应该出现什么结果
- 完成标准：什么结果算完成
- 失败处理：验证失败时进入 `diagnose`

## 标准任务模板

复杂任务优先使用这个结构：

```text
### 任务 N：名称

目标：
文件：
接口：
- 依赖：
- 产出：
测试：
实现步骤：
验证命令：
预期失败：
预期通过：
完成证据：
失败处理：
```

不要写“补充合适测试”“处理边界情况”这类空泛步骤。每个任务都要写清具体行为、具体文件、具体验证命令、预期失败和预期通过。

## 任务粒度

每个任务应该小到可以独立执行、独立测试、独立审查。

好的任务通常包含一个完整测试循环：

1. 写失败测试
2. 运行并确认失败原因正确
3. 写最小实现
4. 运行验证
5. 必要时重构
6. 提交或记录完成状态

不要把任务写成“实现后端”“实现前端”“补测试”这类按技术层横向拆分的任务。

计划应让一个不了解当前会话的人也能执行：写清文件路径、相关接口、验证命令、依赖前置任务和验收证据。

每个任务都必须包含足够的前置上下文：相关领域术语、已有约定、依赖的前置任务、关键接口名称，以及为什么按这一层任务交付。不要依赖当前会话记忆。

## 计划头部

多任务计划必须先写：

- 目标：一句话说明要交付什么用户或系统可观察结果
- 背景：需求来源、关键约束、明确不做什么
- 架构：2-3 句话说明主要实现方向和边界
- 全局验证：完成整组任务后运行哪些命令

不要让执行者从任务列表里反推目标。

## 步骤要求

任务里的步骤必须能执行，不要只写意图。

好的步骤包含：

- 写哪个失败测试
- 运行哪个命令确认失败
- 失败时应看到什么关键信息
- 写哪一处最小实现
- 运行哪个命令确认通过
- 通过时应看到什么关键信息
- 需要重构时如何确认行为不变

如果一个步骤需要代码、命令或文件路径，必须写具体内容。

## 计划检查

- 是否按行为拆分，而不是按技术层拆分？
- 每个任务完成后是否能运行？
- 每个任务是否能被测试证明？
- 是否避免了半成品任务？
- 是否有清晰的验证命令？
- 是否写明失败测试的预期失败结果？
- 是否写明实现后的预期通过结果？
- 后续任务依赖的接口是否已经在前置任务中定义？
- 是否覆盖了所有需求和明确不做事项？
- 是否没有 TODO、占位词或“类似上一步”这类省略？
- 前后任务中的接口名称、文件路径和命令是否一致？
- 是否能交给不了解当前会话的智能体独立执行？
- 是否说明了为什么这样拆，而不是按技术层横向拆？

## 下一步

计划确认后，进入 `execute` 逐项执行任务。
