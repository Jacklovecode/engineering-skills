# TDD

## 实现检查点

凡是新增功能、修复缺陷、重构、行为变更，都默认使用 TDD。

除非用户明确要求跳过，或任务属于纯文档、纯配置、一次性原型，否则不要先写实现代码。

没有先看到失败测试，就不要声称正在做 TDD。

如果已经写了实现但没有失败测试，停止扩展实现。优先撤回或隔离这段实现，再用失败测试重新驱动；如果无法撤回，至少不要让它决定测试写法。

## 红灯-绿灯-重构约束

- 一次只写一个行为测试。
- 测试必须描述外部可观察行为，而不是内部实现细节。
- 优先通过公共接口测试。
- 避免测试私有方法、内部调用次数或临时数据形状。
- 只有在跨进程、网络、时间、随机性或昂贵外部依赖无法真实运行时，才使用测试替身（mock/stub）。
- 先确认公共接口和关键行为；如果接口难以测试，先调整边界，不要为了难测设计而勉强写出脆弱测试。
- 写完测试后必须运行，并确认它因为预期原因失败。
- 只写让当前测试通过的最小实现。
- 测试通过后才能重构。
- 重构期间不得改变行为。
- 每一层任务完成后运行相关验证。
- 如果验证出现非预期失败，进入 `diagnose`，不要直接乱修。

## 公共接口检查

写测试前先确认：

- 测试是否通过用户或系统能观察到的入口验证行为？
- 测试是否避免绑定私有方法、内部调用顺序或临时数据结构？
- 如果重构内部实现但行为不变，测试是否仍应该通过？
- 如果很难通过公共接口测试，是否说明接口边界需要调整？

## 停止标志

出现以下想法或行为时，停止并回到红灯：

- 先写实现，稍后补测试。
- 测试一写就通过，但你无法证明它锁住了新行为。
- 为了让测试通过而修改测试意图。
- 只做手动验证，不留下可重复测试。
- 因为任务很小、时间很急、已有代码无测试，就想跳过 TDD。
- 测试只验证测试替身（mock/stub）、私有方法或内部调用次数。
- 重构时顺手改变行为。

这些都是跳过流程的标志。除非用户明确要求跳过，否则先写能失败的行为测试。

## 常见借口

| 借口 | 现实 |
| --- | --- |
| 这个改动太小 | 小改动也会破坏行为，测试可以很小。 |
| 我先实现再补测试 | 补测容易验证已写实现，而不是验证需求。 |
| 手动试过了 | 手动验证不可重复，不能防回归。 |
| 现有代码没有测试 | 当前改动正是补上行为保护的机会。 |
| 测试太难写 | 难测试通常说明接口边界需要调整。 |

## 任务分层

每轮 TDD 只处理一层可观察行为。不要先铺所有数据库、所有 API 或所有 UI。

一层任务必须能运行、能测试、能被用户或系统观察到。

## 如果已经先写了实现

如果你已经先写了实现代码，但还没有失败测试：

- 不要继续扩展该实现。
- 先停下，补写能失败的行为测试。
- 必要时撤回或隔离先写的实现，避免测试反过来迎合实现。
- 如果无法让测试失败，说明测试没有锁定新行为，必须重写测试。

## 循环

1. 红灯：写一个最小失败测试。
2. 确认红灯：运行测试，确认失败原因正确。
3. 绿灯：写最小实现。
4. 确认绿灯：运行测试和相关验证。
5. 重构：在保持测试通过的前提下清理代码。
6. 下一个行为重复循环。

## 重构方向

绿灯后再重构。优先：

- 去除重复
- 改善命名
- 收紧模块边界
- 把复杂实现藏到简单接口后面
- 删除投机功能

重构后必须重新运行相关测试。

## 下一步

实现完成后进入 `review`。准备声称完成前进入 `finish`。
