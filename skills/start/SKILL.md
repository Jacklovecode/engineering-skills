# Start

## 启动检查点

如果任务明显匹配某个技能，你必须先启用该技能，再继续行动。

如果匹配不确定，可以简短说明你判断的技能路径，然后继续处理低风险任务。

用户的目标、范围和明确边界优先；但用户要求跳过验证、澄清、诊断、审查、完成证据或破坏性操作确认时，不能覆盖这些质量检查点。

## 路由

| 用户意图 | 使用技能 |
| --- | --- |
| 不熟悉代码区域、模块关系不清、需要理解系统全局或影响范围 | `zoom-out` |
| 改进架构、寻找重构机会、降低耦合、提升模块深度、改善可测试性 | `deepen` |
| 新功能、行为变更、架构变更、需求不清 | `clarify` |
| 已有想法、方案、设计或计划，但需要压力测试、拆清边界、完善验收 | `grill` |
| 已有需求但缺少实现路径，或任务跨多个文件/模块 | `plan` |
| 已有确认计划、任务清单或分层任务，需要逐项落地和验证 | `execute` |
| 功能实现、缺陷修复、重构、行为变更 | `tdd` |
| 测试失败、构建失败、行为异常、性能退化、验证无法解释 | `diagnose` |
| 完成实现后、合并前、重大变更后 | `review` |
| 声称完成、已修复、可合并、准备 PR 前 | `finish` |
| 提交、推送、拉取、合并、变基、创建 PR、设置远程、清理分支或 worktree | `git` |
| 创建或修改技能 | `skill-edit` |

## 不要做

- 不要在 `start` 里写代码。
- 不要在 `start` 里诊断根因。
- 不要在 `start` 里写完整计划。
- 不要把“用户明确指令优先”理解成可以跳过质量检查点。

## 输出

保持简短：说明匹配到哪个技能和原因，然后进入对应技能。
