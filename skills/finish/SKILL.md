# Finish

## 完成检查点

在说“完成”“已修复”“可以合并”“测试通过”之前，必须先执行 `finish`。

没有最新验证证据，不允许声明完成。

所有完整路径最终都必须在 `finish` 做最终验收：确认结果是不是用户真正需要的，确认代码、文档或其他产物是不是正确的。

## 证据先行

在作出任何完成判断前，必须先完成：

1. 识别：什么命令、审查或检查能证明该判断？
2. 执行：运行完整验证，不使用过期结果。
3. 读取：查看退出码、失败数量、关键输出和异常信息。
4. 判断：证据是否支持完成结论。
5. 表达：只陈述证据已经证明的内容。

不要使用“应该可以”“看起来没问题”“大概通过”这类没有证据的完成表述。

不要把部分验证说成完整验证。只运行了局部测试，就只能说局部测试通过；没有运行构建、类型检查或端到端验证时，必须说明未验证范围和风险。

## 最终验收点

必须回答：

- 结果是否满足原始需求、计划或用户最新确认的目标？
- 是否遗漏了关键边界、异常路径、验收标准或用户明确约束？
- 产物是否正确？
  - 代码：行为、测试、构建、类型检查、lint 和边界条件是否有证据支持？
  - 文档：内容是否准确、完整、一致，是否避免误导、过期信息和未确认假设？
  - 技能：触发条件、约束、流程、回退路径和验证要求是否自洽？
- 是否实现了未要求的范围，或改变了不该改变的行为？
- 是否还有必须告知用户的残余风险？

## 最终检查

必须检查：

- 需求或规格说明是否完成
- 严重/主要审查问题是否清零
- 相关测试是否通过
- 类型检查、构建和 lint 是否按项目要求通过
- 是否有未处理的调试代码、临时文件、TODO
- 是否有残余风险需要说明

可使用只读脚本辅助收集完成前证据线索：

```text
skills/finish/scripts/collect-evidence.ps1
skills/finish/scripts/collect-evidence.sh
```

脚本只收集状态和推荐验证命令，不代表验证已经通过。必须实际运行验证命令并读取结果。

## 分支和工作区检查

如果当前目录是 git 仓库，完成前检查基础状态；需要提交、推送、合并、创建 PR、清理或回滚时，进入 `git`。

- 当前分支或游离 HEAD 状态（detached HEAD）
- 是否在 git worktree 中
- 工作区是否有未提交改动
- 和对比分支的差异范围

根据状态给出下一步选项：

- 创建 PR
- 本地合并
- 保留当前分支
- 清理 worktree
- 放弃本次改动

不要在用户未确认时执行破坏性清理。

## 完成选项要求

输出下一步选项时，不要只问“接下来做什么”。必须给出结构化选项：

1. 创建 PR：保留当前工作区，便于继续处理反馈。
2. 本地合并：先确认对比分支，再合并，合并后重新验证。
3. 保留当前状态：说明分支、路径和未提交改动。
4. 清理或放弃：列出会删除的内容，并等待明确确认。

如果当前不是 git 仓库，只输出适用于当前环境的交付、保留或清理选项。

如果要放弃或清理工作，必须先列出会删除的分支、提交、文件或 worktree，并等待用户明确确认。

如果要本地合并，必须先验证当前分支，再在合并后重新验证合并结果。

如果要创建 PR，不要清理当前工作区；用户可能还需要继续处理反馈。

如果要清理或放弃，必须把“会删除什么”和“会保留什么”分开列出；没有用户明确确认前，不要删除分支、提交、文件或 worktree。

需要执行具体版本控制操作时，不要在 `finish` 内直接操作，进入 `git`。

## 失败处理

如果最终验证失败、行为异常、结果无法解释：

- 不要直接修。
- 进入 `diagnose`。
- 找到根因并修复后，重新执行 `review` 和 `finish`。

## 输出必须包含

- 完成摘要
- 最终验收结论：结果是否是用户需要的，产物是否正确
- 验证证据：运行了哪些命令，结果是什么
- 审查状态：是否有阻塞项
- 残余风险
- 下一步选项：创建 PR / 合并 / 保留 / 清理

如果某项无法检查，必须说明原因和风险，不要省略。

## 不要做

- 不要在没有验证命令的情况下声称通过。
- 不要把“看起来没问题”当作证据。
- 不要隐藏失败、跳过测试或省略残余风险。
- 不要把部分验证说成完整验证。
- 不要在未确认时删除分支、提交、文件或 worktree。
