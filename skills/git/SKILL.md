# Git

## Git 门禁

涉及版本控制操作时，必须先使用本技能。

包括：

- 初始化仓库
- 查看状态和差异
- 暂存、提交、打标签
- 设置或修改 remote
- fetch、pull、push
- merge、rebase、cherry-pick
- 创建 PR 或准备发布
- 清理分支、提交、文件、stash 或 worktree
- 回滚、重置、强推或改写历史

不要把 git 操作当成普通命令直接执行。先确认状态、范围和风险。

## 上传前检查

提交或推送前必须检查：

- 当前分支和远程跟踪关系。
- 工作区是否有未提交改动。
- 暂存区包含哪些文件。
- 是否误包含本地报告、临时文件、备份文件、密钥、token、本机绝对路径或不应上传目录。
- `.gitignore` 是否覆盖本地生成物。
- 验证命令是否已经运行并通过。

如果用户明确说某个目录不要上传，必须确认：

```text
git ls-files | 匹配该目录
git status --ignored
```

本地保留但不上传时，使用 `.gitignore` 和 `git rm --cached`，不要删除用户本地文件。

## 提交流程

提交前：

1. 运行项目要求的验证命令。
2. 检查 `git status --short --ignored`。
3. 检查 `git diff --cached --check`。
4. 检查暂存文件清单。
5. 确认没有用户明确排除的路径。

提交信息要说明真实改动，不要用空泛描述。

提交后：

- 重新检查状态。
- 如果准备推送，先确认 remote 和远程分支。

## 推送流程

推送前必须确认：

- remote URL 是否是用户指定仓库。
- 远程分支是否已有提交。
- 本地和远程是否有共同历史。
- 是否会覆盖远程历史。

如果远程已有提交：

- 先 `fetch`。
- 比较本地和远程提交。
- 必要时合并或变基。
- 不要直接强推。

只有用户明确要求并理解风险时，才允许强推。

## 合并和变基

合并或变基前：

- 确认当前分支。
- 确认工作区干净，或说明未提交改动。
- 说明将合并或变基的来源分支。

冲突处理：

- 读取冲突内容。
- 保留用户已有意图。
- 解决后运行验证。
- 不要用 `checkout --` 或 `reset --hard` 粗暴覆盖。

## 清理和回滚

以下操作属于破坏性操作：

- 删除分支
- 删除 tag
- 删除 stash
- 删除 worktree
- `git reset --hard`
- `git clean`
- 强推
- 覆盖远程分支
- 删除未提交文件

执行前必须列出会影响的分支、提交、文件或 worktree，并等待用户明确确认。

如果只是让文件不再被 git 跟踪，优先使用：

```text
git rm --cached
```

不要删除本地文件，除非用户明确要求删除本地文件。

## PR 和发布

创建 PR 或发布前：

- 先进入 `finish` 完成最终验收。
- 确认验证证据。
- 确认审查状态。
- 确认没有不应上传内容。
- 确认目标分支和发布范围。

如果 `finish` 发现验证失败或结果无法解释，进入 `diagnose`，不要继续推送或创建 PR。

## 输出必须包含

- 当前分支和 remote。
- 工作区状态。
- 将提交或推送的文件范围。
- 不上传路径检查结果。
- 已运行的验证命令和结果。
- 风险和是否需要用户确认。
- 下一步 git 操作。

## 不要做

- 不要在未检查状态时提交或推送。
- 不要把本地报告、备份、密钥或用户排除目录上传。
- 不要在未确认时删除分支、提交、文件或 worktree。
- 不要在未确认时强推。
- 不要用破坏性命令解决普通冲突。
- 不要把验证失败的代码推送成“完成”。
